## Summary
`GET /admin/users/levels` always returns `500`. The query selects `u.profile`,
but the `users` table has no `profile` column — it has `role` (+ `role_id`,
`level` added via ALTER). The Tech Admin → Staff Hierarchy screen, which lists
users and their levels, is therefore broken.

## Steps to reproduce
1. Log in as admin (L1).
2. `GET /admin/users/levels`.
3. Response is `500 {"error":"... no such column: u.profile ..."}`.

## Root cause
`admin_users_levels` (server/src/lib.rs):
`SELECT u.id, u.username, u.name, u.profile, u.level, ... FROM users u ...`
`u.profile` does not exist (`CREATE TABLE users(... role TEXT, name TEXT)`).

## Resolution (2026-06-27)
Changed the column to `u.role AS profile`, so the JSON key stays `profile`
(matching `/auth/login`, which already returns the role under `profile`). The
endpoint now returns 200 with the user list.

## Regression coverage
`tests/api/permissions.spec.ts` → "L1 admin can reach admin routes" hits
`/admin/users/levels` and asserts 200, so this can't silently regress.
